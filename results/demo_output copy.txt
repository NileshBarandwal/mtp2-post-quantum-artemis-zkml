  This demo proves the following thesis claim:

    KZG polynomial commitments — as used in the Artemis zkML
    framework — are broken by Shor's algorithm (demonstrated
    classically via BSGS). Replacing them with SHA-256 hash
    commitments eliminates all three ECC-dependent attack
    surfaces and restores post-quantum security.

  Note           :  Group non-cyclic (1004=4×251); max element order=502
  Weight w=42    :  valid scalar (42 < n=502) ✓

  Shor's complexity: O((log n)³) — polynomial in key size

  Conclusion     :  ECDLP provides NO quantum security

  ════════════  KZG COMMITMENT — FULL ATTACK CHAIN  ════════════

  CONTEXT: Artemis CP-SNARK for zkML
  An ML model owner wants to prove inference is correct
  WITHOUT revealing the model weights to the verifier.

  CLAIM: KZG commitments provide this guarantee.
  ATTACK: A quantum adversary using Shor's algorithm breaks it.
  FIX:    Replace KZG with a hash-based commitment (Session 3).

  ═══  PHASE A — Honest Commitment  (PC.Setup + PC.Commit)  ════

  The model owner generates the SRS and commits to their weight.

  ────────────────────────────────────────────────────────────
  [SLIDE 13 — STEP 1]  Generate secret τ  (the KZG trapdoor)

    τ generated    (value kept secret — not printed)
    τ is a random scalar in {2, ..., n-1} = {2, ..., 501}
    Time           : 2.71 µs

  [SLIDE 13 — STEP 2]  Compute powers of τ on the curve

    Computing SRS = [τ⁰G, τ¹G, τ²G]  (degree-2 SRS)
    Each τⁱG is a scalar multiplication on y²=x³+2x+3 mod 1021

    SRS[0] = τ^0·G = 1·G = (0, 989)

  ═  PHASE B — Quantum Attack  (BSGS = classical Shor's equivalent)  ═

  An adversary with a quantum computer sees the published commitment C
  and the public SRS. They run Shor's algorithm (simulated here via BSGS)
  to recover the secret weight w.

  Adversary has   :  C = (1019, 920),  G = (0, 989),  n = 502
  Adversary wants :  w  such that  C = w·G

  [BSGS] Setting m = ⌈√502⌉ = 23
  [BSGS] Baby steps  : {i·G → i}  for i = 0..22
  [BSGS] Giant steps : C - j·(m·G)  for j = 0..23

  ┌─ ATTACK RESULT ──────────────────────────────────────────────
  │  Recovered w   : 42  ← SECRET WEIGHT EXPOSED
  │  Verified      : 42·G = (1019, 920) = C ✓
  │  Attack time   : 0.0222 ms
  │
  │  CONSEQUENCE: The model owner's private weight is now known.
  │  Artemis privacy guarantee — BROKEN.
  └──────────────────────────────────────────────────────────────

  Attacker now knows: the neural network y = 42·x + 7
  This is the exact model that was committed. Privacy is GONE.

  ══  PHASE C — Commitment Forgery  (fraudulent model w'=99)  ══

  The attacker now creates a FAKE commitment to a DIFFERENT weight.
  Goal: convince the verifier that the committed model uses w'=99,
  not the true weight w=42.

  This would let the attacker:
    • Sell a high-quality model (w=42) but prove ownership of garbage (w'=99)
    • Submit proofs of correct inference for a model they don't own
    • Undermine any zkML marketplace built on Artemis

  True commitment  :  C_real = 42·G = (1019, 920)   [w = 42]
  Forged commitment:  C_fake = 99·G = (736, 325)  [w'= 99]

  C_fake is on curve :  True ✓
  C_fake ≠ C_real    :  True ✓  (genuinely different weight)
  Forge time         :  5.13 µs  (trivial once w is known)

  The verifier runs PC.Verify on both the real and forged commitments.
  KZG verification only checks: 'does w_claimed·G equal C?'
  It CANNOT detect forgery — any valid (w', C') pair will pass.

  ┌─ BEFORE ATTACK — Honest verification ────────────────────────
  │  Verifier checks: does the prover know w such that w·G = C_real?
  │  C_real = (1019, 920)
  │
  │  Claimed w = 42   →   42·G = (1019, 920)
  │  C_real    = (1019, 920)

  The verifier CANNOT DISTINGUISH real from forged.

  Step 1  [COMMIT]    Model owner publishes C = 42·G = (1019, 920)
  Step 2  [ATTACK]    Adversary runs BSGS on C → recovers w = 42

  │  Verify (fake) :  5.17 µs
  └───────────────────────────────────────────────────────────────

  ROOT CAUSE: KZG commitment C = w·G leaks group structure.
  ECDLP protects w classically but NOT against Shor's algorithm.
  All three ECC dependencies in Artemis trace to this single weakness:
    1. SRS generation   → τ can be recovered from τG via Shor's
    2. Commitment       → w can be recovered from w·G via Shor's
    3. Verifying key    → pairing-based, collapses with ECC

  PROPOSED FIX (Session 3):
    Replace C = w·G  with  C = SHA256(w || nonce)
    BSGS has no group structure to exploit → attack fails by design.

  THREE ECC DEPENDENCIES IN ARTEMIS  (all broken by Shor's):

  1. Structured Reference String (SRS / Trusted Setup)
     Role    :  SRS = [G, τG, τ²G, ...]  is the public parameter
                used by all provers and verifiers.
     Threat  :  Shor's recovers τ from τG = SRS[1].
                Knowing τ allows forging any KZG proof from scratch.
     Fix     :  ELIMINATED — hash commitment needs no SRS or τ.

  2. Polynomial Commitment  (C = w·G)
     Role    :  The model owner publishes C = w·G to commit to
                weight w without revealing it.
     Threat  :  Shor's (/ BSGS classically) recovers w from C and G
                by solving the ECDLP: given C = w·G, find w.
                Session 2 showed this in milliseconds on a small curve.
     Fix     :  Replaced by C = SHA256(w ∥ r).  Shor's has no group
                structure to exploit.  Session 3 confirmed BSGS fails.

  3. Verifying Key  (VK = (G, τG)  — pairing-based)
     Role    :  The verifying key enables the verifier to check proofs
                without running the full prover computation.
     Threat  :  VK is ECC-based (BN254 pairing).  Shor's breaks the
                underlying discrete log → VK can be forged or bypassed.
     Fix     :  Replaced by nonce r (32 bytes).  Verification is just
                SHA256(w_claimed ∥ r) == C — no pairings, no ECC.
